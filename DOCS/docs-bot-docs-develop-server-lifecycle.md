<h1>Server lifecycle | Root developer documentation</h1>
<header><h1>Server lifecycle</h1></header>
<p>Your server runs in the Root cloud, and Root manages when it starts, stops, and restarts. This article explains how the server lifecycle works, how to define your entry point, run initialization code, and handle orderly shutdowns.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="when-does-your-server-run">When does your server run?<a href="#when-does-your-server-run" class="hash-link" aria-label="Direct link to When does your server run?" title="Direct link to When does your server run?" translate="no">​</a></h2>
<p>Root controls when your server runs. The two normal cases are:</p>
<ul>
<li class="">When your app is installed into a community, Root creates and starts an instance for that community.</li>
<li class="">Root might restart your server during maintenance.</li>
</ul>
<p>There are three situations where Root will automatically restart your server after a problem:</p>
<ul>
<li class="">If your server crashes.</li>
<li class="">If you intentionally stop your server.</li>
<li class="">If there's an unhandled exception during startup, shutdown, a scheduled job, or inside your Community API event handler.</li>
</ul>
<p>When your server restarts, all attached clients in that community automatically get restarted.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="when-will-root-take-your-server-offline">When will Root take your server offline?<a href="#when-will-root-take-your-server-offline" class="hash-link" aria-label="Direct link to When will Root take your server offline?" title="Direct link to When will Root take your server offline?" translate="no">​</a></h2>
<p>Root may take your server offline if Root determines it's delivering a poor customer experience. Generally, this occurs when your server crashes repeatedly.</p>
<p>Servers are taken offline on a community-by-community basis. That is, if your server crashes repeatedly in one community, it's only taken offline in that individual community.</p>
<p>You'll need to release an updated version of your code to re-enable your server.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="what-is-the-server-lifecycle">What is the server lifecycle?<a href="#what-is-the-server-lifecycle" class="hash-link" aria-label="Direct link to What is the server lifecycle?" title="Direct link to What is the server lifecycle?" translate="no">​</a></h2>
<p>The <em>lifecycle</em> is the set of states your server moves through: <em>NotRunning</em>, <em>Starting</em>, <em>Started</em>, <em>Stopping</em>, and <em>Stopped</em>.</p>
<!-- -->
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="what-do-you-control">What do you control?<a href="#what-do-you-control" class="hash-link" aria-label="Direct link to What do you control?" title="Direct link to What do you control?" translate="no">​</a></h2>
<p>You control two things as your server moves through its lifecycle:</p>
<ol>
<li class="">Required: call <code>rootServer.lifecycle.start</code> to move from <strong>NotRunning</strong> to <strong>Starting</strong>.</li>
<li class="">Optional: call <code>rootServer.lifecycle.stop</code> to move from <strong>Started</strong> to <strong>Stopping</strong> if your code can’t continue.</li>
</ol>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="lifecycle-callbacks">Lifecycle callbacks<a href="#lifecycle-callbacks" class="hash-link" aria-label="Direct link to Lifecycle callbacks" title="Direct link to Lifecycle callbacks" translate="no">​</a></h2>
<p>The Root API lets you register three lifecycle callbacks that Root invokes when your server enters the corresponding state. They're all optional.</p>
<ol>
<li class="">A <strong>starting callback</strong> to handle initialization during the <strong>Starting</strong> state.</li>
<li class="">A <strong>stopping callback</strong> to begin your cleanup during the <strong>Stopping</strong> state.</li>
<li class="">A <strong>stopped callback</strong> to do any final cleanup during the <strong>Stopped</strong> state.</li>
</ol>
<!-- -->
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="starting">Starting<a href="#starting" class="hash-link" aria-label="Direct link to Starting" title="Direct link to Starting" translate="no">​</a></h3>
<p>Root invokes your <strong>starting callback</strong> when your server moves into the <strong>Starting</strong> state. Do all your initialization and setup here; Root can’t guarantee the Root APIs are ready before this point.</p>
<p>Typical things you might do here:</p>
<ul>
<li class="">Register services.</li>
<li class="">Initialize database.</li>
<li class="">Apply global settings.</li>
<li class="">Subscribe to API events.</li>
</ul>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="stopping">Stopping<a href="#stopping" class="hash-link" aria-label="Direct link to Stopping" title="Direct link to Stopping" translate="no">​</a></h3>
<p>Root invokes your <strong>stopping callback</strong> when your server moves into the <strong>Stopping</strong> state. This state means Root is giving you early notice that it has begun the process of shutting down your server.</p>
<p>You should stop any long-running process and persist any unsaved state. If you need to store anything external to Root, you should do it here so you have the most time to get it done. You might also consider logging a message either to yourself by printing to the console or to the community using the logging API.</p>
<p>If your code saves state aggressively as it runs, you might not need to register this callback.</p>
<p>Your stopping callback will run only in cases of orderly shutdown like when Root maintenance causes a restart of your server or you call <code>rootServer.lifecycle.stop</code>. It isn't guaranteed to run during abnormal termination; for example, if your server crashes.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="stopped">Stopped<a href="#stopped" class="hash-link" aria-label="Direct link to Stopped" title="Direct link to Stopped" translate="no">​</a></h3>
<p>Root invokes your <strong>stopped callback</strong> when your server moves into the <strong>Stopped</strong> state. This state means Root has begun the process of shutting down the cloud container that's running your server. This is your final opportunity to do any cleanup or save state.</p>
<p>As with the <strong>stopping callback</strong>, if your code saves state aggressively as it runs, you might not need this callback.</p>
<p>Your stopped callback will run only in cases of orderly shutdown like when Root maintenance causes a restart of your server or you call <code>rootServer.lifecycle.stop</code>. It isn't guaranteed to run during abnormal termination; for example, if your server crashes.</p>
